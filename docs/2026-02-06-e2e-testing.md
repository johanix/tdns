# End-to-End Testing Guide

**Date:** 2026-02-06
**Status:** Implemented
**Related Issues:** DNS-57, DNS-64

## Overview

This guide provides comprehensive testing procedures for the DNS CHUNK NOTIFY transport system, including test scenarios, validation criteria, and troubleshooting guidance.

## Test Environment Setup

### Prerequisites

1. **Multiple Agent Instances:**
   - Minimum: 2 agents (alpha, bravo)
   - Recommended: 3-4 agents for full mesh testing
   - Optional: 1 combiner instance

2. **DNS Infrastructure:**
   - Each agent needs DNS resolution
   - Agents must be able to query each other
   - Valid JWK records published for each agent

3. **Configuration:**
   - Agent DNS transport enabled: `agent.dns.enabled: true`
   - Message retention configured (optional but recommended)
   - Router introspection enabled

### Starting Test Agents

```bash
# Agent Alpha
tdns -c agent-alpha.yaml &

# Agent Bravo
tdns -c agent-bravo.yaml &

# Agent Charlie (optional)
tdns -c agent-charlie.yaml &
```

Verify agents are running:
```bash
# Check each agent's status
tdns-cli --api http://localhost:8001 agent status
tdns-cli --api http://localhost:8002 agent status
tdns-cli --api http://localhost:8003 agent status
```

## Test Scenarios

### Scenario 1: Agent Discovery and Hello Exchange

**Goal:** Verify agents can discover each other and establish working keys.

**Test Steps:**

1. **Check initial agent registry:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent dump-agent-registry \
     --zone example.com
   ```

   **Expected:** Empty or minimal registry (only self)

2. **Trigger hello message from alpha to bravo:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent send-notify \
     --zone example.com \
     --id agent.alpha \
     --rrtype A \
     --RR "test.example.com. A 192.0.2.1"
   ```

3. **Verify bravo is registered:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent dump-agent-registry \
     --zone example.com
   ```

   **Expected:**
   - agent.bravo appears in registry
   - Status: active
   - Working Keys: >= 1

4. **Check router metrics:**
   ```bash
   tdns-cli --api http://localhost:8001 agent router metrics
   ```

   **Expected:**
   - Total Messages > 0
   - Hello message handler called
   - No errors

**Success Criteria:**
- ✓ Agents discover each other
- ✓ Working keys established
- ✓ No authorization errors
- ✓ Router metrics show successful hello processing

**Common Issues:**
- DNS resolution failures → Check agent DNS names resolve
- Authorization failures → Verify JWK records published
- No hello messages → Check DNS transport enabled

### Scenario 2: Beat (Heartbeat) Messages

**Goal:** Verify periodic heartbeat messages maintain peer connectivity.

**Test Steps:**

1. **Wait for automatic beat messages** (sent every 30 seconds by default)

2. **Check router metrics:**
   ```bash
   tdns-cli --api http://localhost:8001 agent router metrics
   ```

   **Expected:**
   - Beat handler call count increases over time
   - No errors

3. **Check distribution cleanup:**
   ```bash
   tdns-cli --api http://localhost:8001 agent distrib list
   ```

   **Expected:**
   - Beat distributions expire after 30 seconds
   - List doesn't accumulate excessive beat entries

4. **Force a beat message:**
   ```bash
   # Send beat via debug command (if implemented)
   tdns-cli --api http://localhost:8001 debug agent send-notify \
     --zone example.com \
     --id agent.alpha \
     --rrtype BEAT \
     --RR "beat-test"
   ```

**Success Criteria:**
- ✓ Beat messages sent periodically
- ✓ Beat distributions cleaned up after 30s
- ✓ Peer connections maintained
- ✓ No authorization errors

**Common Issues:**
- Beat accumulation → Check message retention config
- High error counts → Review authorization middleware
- Distributions not cleaning up → Verify GC running

### Scenario 3: Zone Synchronization

**Goal:** Verify zone data synchronizes correctly between agents.

**Test Steps:**

1. **Check initial zone state on both agents:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent dump-zone-data-repo \
     --zone example.com
   tdns-cli --api http://localhost:8002 debug agent dump-zone-data-repo \
     --zone example.com
   ```

   **Expected:** Both show same version number

2. **Trigger sync from alpha:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent trigger-sync \
     --zone example.com \
     --from agent.alpha \
     --rr "test.example.com. A 192.0.2.1"
   ```

   **Expected Output:**
   ```
   Sync triggered successfully:
   Initiated sync from agent.alpha for example.com
   Syncing to N peers: agent.bravo, ...
   ```

3. **Verify sync on bravo:**
   ```bash
   tdns-cli --api http://localhost:8002 debug agent sync-state \
     --zone example.com
   ```

   **Expected:**
   - Version numbers match between agents
   - Sync status shows "synced"

4. **Check distribution status:**
   ```bash
   tdns-cli --api http://localhost:8001 agent distrib list
   ```

   **Expected:**
   - Sync distribution shows as "confirmed"
   - Correlation IDs match

5. **Verify zone data:**
   ```bash
   tdns-cli --api http://localhost:8002 debug agent dump-zone-data-repo \
     --zone example.com
   ```

   **Expected:**
   - New RR appears in recent changes
   - Version incremented

**Success Criteria:**
- ✓ Sync message sent successfully
- ✓ Sync message received and acknowledged
- ✓ Zone versions synchronized
- ✓ RR changes propagated
- ✓ Distribution marked confirmed

**Common Issues:**
- Sync timeout → Check network connectivity
- Version mismatch → Review sync logic
- No acknowledgment → Check router sync handler
- Distribution stuck pending → Verify confirmation processing

### Scenario 4: Multi-Peer Synchronization

**Goal:** Verify synchronization works across multiple peers simultaneously.

**Test Steps:**

1. **Set up 3+ agents** (alpha, bravo, charlie)

2. **Verify all agents discovered:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent dump-agent-registry \
     --zone example.com
   ```

   **Expected:** All peers listed

3. **Trigger multi-peer sync:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent trigger-sync \
     --zone example.com \
     --from agent.alpha \
     --rr "multi.example.com. A 192.0.2.10"
   ```

   **Expected Output:**
   ```
   Syncing to 2 peers: agent.bravo, agent.charlie
   ```

4. **Verify sync on all peers:**
   ```bash
   # Check bravo
   tdns-cli --api http://localhost:8002 debug agent sync-state \
     --zone example.com

   # Check charlie
   tdns-cli --api http://localhost:8003 debug agent sync-state \
     --zone example.com
   ```

   **Expected:** All show same version, all synced

5. **Check router metrics on all agents:**
   ```bash
   tdns-cli --api http://localhost:8001 agent router metrics
   tdns-cli --api http://localhost:8002 agent router metrics
   tdns-cli --api http://localhost:8003 agent router metrics
   ```

   **Expected:**
   - Alpha: Sync handler call count = 0 (sender)
   - Bravo: Sync handler call count >= 1
   - Charlie: Sync handler call count >= 1

**Success Criteria:**
- ✓ Sync reaches all peers
- ✓ All peers acknowledge
- ✓ Versions synchronized across mesh
- ✓ No peers left behind

**Common Issues:**
- Partial sync → Check per-peer connectivity
- One peer lagging → Force sync to that peer
- High latency → Review network conditions

### Scenario 5: Force Sync to Specific Peer

**Goal:** Verify ability to force synchronization with a specific peer.

**Test Steps:**

1. **Identify out-of-sync peer:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent sync-state \
     --zone example.com
   ```

   **Expected:** Shows which peers are behind

2. **Force sync to specific peer:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent force-sync \
     --zone example.com \
     --peer agent.bravo \
     --rr "force.example.com. A 192.0.2.20"
   ```

   **Expected Output:**
   ```
   Force sync completed:
   Successfully synced with agent.bravo
   Correlation ID: corr-xxxxx
   ```

3. **Verify peer synchronized:**
   ```bash
   tdns-cli --api http://localhost:8002 debug agent sync-state \
     --zone example.com
   ```

   **Expected:** Version matches source agent

**Success Criteria:**
- ✓ Targeted peer receives sync
- ✓ Other peers not affected
- ✓ Version gap closed

**Common Issues:**
- Peer still behind → Check correlation ID in distributions
- Force sync fails → Verify peer online and reachable

### Scenario 6: Full Test Chain

**Goal:** Execute end-to-end test of complete sync pipeline.

**Test Steps:**

1. **Run test chain with add scenario:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent test-chain \
     --zone example.com \
     --scenario add \
     --rr "chain-test.example.com. A 192.0.2.30"
   ```

   **Expected Output:**
   ```
   Test Chain Results:
   ===================

   Scenario: add
   Zone: example.com
   RRs Count: 1

   Step 1 (Local Update):
     ✓ Success: Zone data updated locally (version 45)

   Step 2 (Peer Sync):
     Peers synced: 2
       ✓ agent.bravo: Sync completed
       ✓ agent.charlie: Sync completed

   Step 3 (Combiner Processing):
     ⊘ Skipped: Combiner not configured
   ```

2. **Verify all steps succeeded:**
   - Check each step shows ✓
   - Note correlation IDs

3. **Run test chain with update scenario:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent test-chain \
     --zone example.com \
     --scenario update \
     --rr "chain-test.example.com. A 192.0.2.31"
   ```

4. **Run test chain with delete scenario:**
   ```bash
   tdns-cli --api http://localhost:8001 debug agent test-chain \
     --zone example.com \
     --scenario delete \
     --rr "chain-test.example.com. A 192.0.2.31"
   ```

**Success Criteria:**
- ✓ All steps complete successfully
- ✓ Local update succeeds
- ✓ Peer sync succeeds for all peers
- ✓ Combiner processes (if configured)

**Common Issues:**
- Step 1 fails → Check zone configuration
- Step 2 fails → Review peer connectivity
- Step 3 skipped → Expected if no combiner

### Scenario 7: Message Retention and Cleanup

**Goal:** Verify distribution cleanup based on message type retention.

**Test Steps:**

1. **Check current distributions:**
   ```bash
   tdns-cli --api http://localhost:8001 agent distrib list
   ```

   **Expected:** Mix of message types

2. **Send beat message:**
   ```bash
   # Beat messages expire after 30s
   tdns-cli --api http://localhost:8001 debug agent send-notify \
     --zone example.com \
     --id agent.alpha \
     --rrtype BEAT \
     --RR "test-beat"
   ```

3. **Wait 35 seconds**

4. **Check distributions again:**
   ```bash
   tdns-cli --api http://localhost:8001 agent distrib list
   ```

   **Expected:** Beat distribution removed (30s expiration)

5. **Send sync message:**
   ```bash
   # Sync messages expire after 300s (5 min)
   tdns-cli --api http://localhost:8001 debug agent trigger-sync \
     --zone example.com \
     --from agent.alpha \
     --rr "test A 1.2.3.4"
   ```

6. **Wait 35 seconds**

7. **Check distributions:**
   ```bash
   tdns-cli --api http://localhost:8001 agent distrib list
   ```

   **Expected:** Sync distribution still present (300s retention)

**Success Criteria:**
- ✓ Beat distributions expire after 30s
- ✓ Sync distributions retained for 5 min
- ✓ GC runs automatically
- ✓ No memory leaks from unbounded accumulation

**Common Issues:**
- Distributions not expiring → Check GC goroutine running
- Wrong expiration times → Verify config settings
- All distributions expire immediately → Check ExpiresAt calculation

## Validation Criteria

### Per-Test Validation

After each test scenario, verify:

1. **No Error Logs:**
   ```bash
   # Check agent logs for errors
   grep ERROR /var/log/tdns-agent-alpha.log
   ```

2. **Router Metrics:**
   - Handler errors should be 0
   - Middleware errors should be 0
   - Unknown messages should be 0

3. **Distribution Status:**
   - All distributions either confirmed or expired
   - No stuck "pending" distributions beyond timeout

4. **Memory Usage:**
   - No unbounded growth
   - Distributions cleaned up properly

### Regression Testing

Run all scenarios after any code changes to DNS transport system:

```bash
# Automated regression test script
./test/e2e/run-all-scenarios.sh
```

**Expected:** All scenarios pass without errors.

## Troubleshooting Guide

### Issue: Authorization Failures

**Symptoms:**
- Router metrics show middleware errors
- Logs contain "authorization failed"
- Messages not reaching handlers

**Debug Steps:**
1. Check JWK records published:
   ```bash
   dig +short _jwk.agent-alpha.example.com TXT
   ```

2. Verify working keys established:
   ```bash
   tdns-cli --api http://localhost:8001 debug agent dump-agent-registry \
     --zone example.com
   ```

3. Check peer key cache

**Solutions:**
- Re-publish JWK records
- Trigger hello exchange
- Clear key cache and retry

---

### Issue: Sync Not Completing

**Symptoms:**
- Distributions stuck in "pending"
- Versions don't match between agents
- Timeout errors

**Debug Steps:**
1. Check distribution status:
   ```bash
   tdns-cli --api http://localhost:8001 agent distrib list
   ```

2. Review sync state:
   ```bash
   tdns-cli --api http://localhost:8001 debug agent sync-state \
     --zone example.com
   ```

3. Check router metrics for errors:
   ```bash
   tdns-cli --api http://localhost:8001 agent router metrics
   ```

4. Force sync to problem peer:
   ```bash
   tdns-cli --api http://localhost:8001 debug agent force-sync \
     --zone example.com \
     --peer agent.problematic
   ```

**Solutions:**
- Check network connectivity
- Verify peer is online
- Review correlation IDs for tracking
- Check for packet loss

---

### Issue: Router Not Initialized

**Symptoms:**
- API errors: "Router not available"
- Router commands fail

**Debug Steps:**
1. Check agent config:
   ```yaml
   agent:
     dns:
       enabled: true  # Must be true
   ```

2. Verify agent started without errors:
   ```bash
   grep "Router initialized" /var/log/tdns-agent.log
   ```

**Solutions:**
- Enable DNS transport in config
- Restart agent
- Check for initialization errors in logs

---

### Issue: High Memory Usage

**Symptoms:**
- Agent memory grows over time
- Distribution list very long
- System slowdown

**Debug Steps:**
1. Check distribution count:
   ```bash
   tdns-cli --api http://localhost:8001 agent distrib list | wc -l
   ```

2. Check message retention config

3. Verify GC running:
   ```bash
   grep "Distribution GC: purged" /var/log/tdns-agent.log
   ```

**Solutions:**
- Reduce retention times in config
- Verify GC goroutine started
- Manually purge old distributions:
  ```bash
  tdns-cli --api http://localhost:8001 agent distrib purge
  ```

## Performance Testing

### Load Testing

Send high volume of messages to test router performance:

```bash
# Send 1000 sync messages
for i in {1..1000}; do
  tdns-cli --api http://localhost:8001 debug agent trigger-sync \
    --zone example.com \
    --from agent.alpha \
    --rr "load-test-$i.example.com. A 192.0.2.$((i % 256))"
  sleep 0.1
done
```

**Monitor:**
- Router metrics (call counts, latencies)
- Distribution cleanup effectiveness
- Memory usage
- CPU usage

### Latency Testing

Measure end-to-end sync latency:

```bash
# Time a sync operation
time tdns-cli --api http://localhost:8001 debug agent force-sync \
  --zone example.com \
  --peer agent.bravo \
  --rr "latency-test.example.com. A 192.0.2.100"
```

**Target:** < 100ms for local network
